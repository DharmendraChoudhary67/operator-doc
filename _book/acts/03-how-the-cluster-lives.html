The Living Cluster

Now we discuss how the Valkey cluster works at runtime: how the client reaches the correct node, how headless services help with identity, how gossip spreads cluster knowledge, and what changes when pods restart.

Valkey Cluster Basics

A Valkey cluster is not just a set of identical pods. Each member has a role and a relationship to the wider topology.

  • primaries own hash slots
  • replicas shadow those primaries and are candidates for promotion
  • gossip keeps topology and failure knowledge moving between nodes
  • clients need a current slot map so they can reach the right owner

That is why cluster behavior cannot be reduced to “one service in front of many pods.”

How the Client Connects to the Cluster

In a real cluster, the client does not blindly send every request to one random node. A cluster-aware client learns the slot map and then targets the correct primary.

The important point is this:

  • a service or load balancer can provide the first entry point
  • but correct request routing depends on slot ownership
  • after failover, the client must refresh its view and talk to the new owner

Normal Service vs Headless Service

For stateful systems, these two service types do different jobs.

Service type What it gives you Why it matters here
normal Service one stable service identity and generic traffic distribution useful as an entry point
headless Service pod-specific DNS records gives peers and controllers a stable way to address individual members

The headless service is especially important for StatefulSets because it gives each pod a stable DNS identity even when pod IPs change (Pod restart).

How Gossip Drives Failover

Valkey uses its cluster bus to exchange health and topology information between nodes. This is the gossip protocol that makes runtime failure handling self-contained.

The important point is that the operator may help with cluster formation, but normal leader failure detection and replica promotion happen inside the Valkey cluster itself.

In short:

  • Possible Failure (PFAIL) is a local suspicion
  • FAIL is a cluster-level decision
  • promotion needs replica candidacy plus cluster agreement
  • clients see the result through redirects and slot-map refresh, not through the gossip messages directly

What Happens When a Leader or Follower Falls

When a Leader Fails

What changes after a leader failure:

  • failure is first suspected locally, then confirmed by cluster gossip
  • an eligible replica is promoted to the new primary
  • slot ownership moves to the promoted replica
  • clients must refresh routing or follow redirects before writes fully normalize
  • the old primary can later return, but it usually comes back as a replica

One subtle point from the failover notes is important: failover changes roles, not pod placement. The pods usually stay on the same Kubernetes nodes; what changes is the internal Valkey role attached to each pod.

When a Follower Fails

What changes after a follower failure:

  • client write routing usually does not change immediately because the primary still owns the slot
  • replication safety drops because one copy is missing or stale
  • the shard remains functional, but it is less protected until the follower returns or a replacement is attached
  • if the follower comes back with persistent identity, rejoin is smoother; without that identity, recovery becomes more manual

So the networking story is inseparable from topology and failover:

  • the client only sees slot owners and redirects
  • the cluster uses gossip to decide whether a node is actually down
  • safe recovery depends on role transitions, storage identity, and physical separation all working together

A Pod Without Memory

Two situations lead to the same operational outcome:

  • a brand new pod joins the cluster for the first time
  • an old pod restarts without its previous cluster state, usually because nodes.conf and related files were not preserved

In both cases, the cluster sees an unknown member. The pod starts the Valkey process, opens the client and cluster-bus ports, and waits to be introduced.

The important idea is that “pod running” is not the same as “node joined.” Without preserved cluster memory, the pod does not automatically count as the same logical member, even if the container name looks familiar.

The Return With Memory

When the pod restarts with persistent storage and nodes.conf intact, the return path is much smoother.

  • the pod may come back on a new IP
  • nodes.conf restores the previous node ID
  • peers recognize the same member reconnecting
  • the cluster updates the current address for that identity

This is why persistent storage is not just about bytes. It is also about preserving cluster identity.

That difference is one of the central lessons of the whole booklet: a restarted container is not automatically the same cluster member unless the right memory survives with it.
